## P3: Tenancy primitives at enterprise grade

**Goal:** Per-tenant data is real. Architectural commitments inherited from P2 (database-per-tenant per D1, jurisdiction as a column per D12, tenant onboarding as configuration per D13, audit as a bounded context per D22, tenant-isolation tests per D24, configuration through `vadakkan/config/` per D19, envelope encryption per D21) move from charter commitments to running code enforced by tests.

**Scope expansion from `packages.md` original line:** "Tenancy primitives" includes the credential and migration machinery that makes per-tenant data real at enterprise grade. Settled at P3 framing in Claude.ai.

**Sessions in this package:**
- S9: Tenancy context scaffolding. `contexts/tenancy/` with the bounded-context structure. Domain model (Tenant, TenantId, EncryptedCredentials, TenantConnectionConfig). Registry port. Compose stack adds postgres-control-plane plus two per-tenant Postgres instances (postgres-tenant-a, postgres-tenant-b) for the test set. Vadkkan-to-Vadakkan directory housekeeping folded in. D32 (instance topology) and D33 (control-plane separation) appended.
- S10: Control-plane database and registry adapter shipped. Alembic configured for the control-plane track via the named-section pattern (single `alembic.ini`, `[control_plane]` section per the cookbook bracket form). Registry Postgres adapter implements CRUD with envelope encryption (D21) on the credential write/read paths; AAD binds `tenant_id` and a fixed purpose string. The three leak-prevention controls (logging filter, AST test forbidding plaintext-in-state, registry isolation tests) shipped in the same commits as the integration. Use cases enforce operator-context-only access at `reveal_connection_config`. Audit events on registry mutations carry the control-plane sentinel for `tenant_id`. `ops/run_scheduled_checks.py` lands the runner; `make scheduled-check` produces a Markdown report. D34 records the integration shape.
- S11: Per-tenant connection resolution and migration runner shipped. Routing layer landed in `contexts/tenancy/` per D36 (re-homed from the S9-prompt's `vadakkan/config/` placement to keep the dependency graph DAG-shaped): `contexts/tenancy/application/connection_resolution.py` composes `reveal_connection_config` (operator-context system actor) with `contexts/tenancy/adapters/outbound/sqlalchemy/session_factory.py`; per-tenant `AsyncEngine` instances cached keyed on `tenant_id`, invalidated on `update_tenant_status`. Per-tenant Alembic track wired as a second `[tenant]` named section under the existing `alembic.ini` with its own `alembic/tenant/` script directory. `make migrate` runs both phases in order (control-plane first, per-tenant second), per-tenant transactional. Per-tenant initial schema (the `tenant_audit` table per D22) applied to `postgres-tenant-a` and `postgres-tenant-b`. `make seed-tenants` registers the test set idempotently. Host-port-binding rule promoted from compose.yaml comment to AST-level parser test enforcing the loopback-only convention. D35 (audit destination convention) and D36 (routing layer + migration runner) appended.
- S12: Audit context real adapter and end-to-end verification. Audit adapter replaces no-op (since S5) with real Postgres adapter writing hash-chained events to the per-tenant `tenant_audit` table (schema landed at S11 per D36) and to a control-plane audit table (schema lands here), routed by the empty-string-sentinel convention promoted as D35. Per-tenant routing uses `get_tenant_session_factory` from D36's routing layer. Integration test for the full slice: authenticated request → tenant-scoped endpoint → routing → tenant database → hash-chained audit write → Langfuse trace with tenant_id and jurisdiction attributes. Browser verification of trace structure mid-session per the standing rule.
