## Architectural commitments shipped

Six D-entries (D32-D37). Fewer than P2's sixteen because P3 was implementation-heavy on architecture P2 laid down. The structural ones:

- **D32, D33.** Per-tenant Postgres instance topology (separate instances) and control-plane database on a dedicated instance. Database-per-tenant moves from architectural commitment (D1) to running implementation.
- **D34.** Credential encryption integration shape: envelope encryption with AAD binding `tenant_id` + purpose, three leak-prevention controls (logging filter, AST test forbidding plaintext-in-state, tenant-isolation tests) in the same commits as the integration. SQLAlchemy 2.0 Core-not-ORM choice forced by frozen-dataclass aggregate.
- **D35.** Audit destination convention: empty-string `tenant_id` sentinel for control-plane scope; per-destination hash chains; one `AuditPort` with adapter-level routing.
- **D36.** Per-tenant connection routing layer with cached session factories in `contexts/tenancy/`; two-phase migration runner (control-plane first, per-tenant second); per-tenant transactional with retry rather than cross-tenant atomicity.
- **D37.** Audit adapter shape: SELECT FOR UPDATE on chain-tail row inside the same transaction at READ COMMITTED isolation; OTel span attributes `tenant.id` + `tenant.jurisdiction` as Vadakkan-domain; schema-level CHECK constraints enforcing per-destination sentinel/non-sentinel match. Hash-cache deferred per measurement-first.

## Data the package surfaced

- **Memory cost of database-per-tenant topology in dev:** 35-42 MiB per Postgres instance at idle (S9 measurement). Adding three Postgres instances cost ~115 MiB total — an order of magnitude smaller than the inference overhead from S6 (Qwen 2.5 7B at ~5GB resident). The cost calibration argument for D32's separate-instances posture holds at this scale.
- **Test density across P3:** S9 +14, S10 +29, S11 +12, S12 ~+14 = ~+69 tests over the P2-close baseline of 59. Heavier test density than P2's structural sessions because P3 introduces tenant-scoped data, and D24's mandate produces isolation tests for every adapter that touches it.
- **Import-linter contract count:** stable at 15 (one new contract in S9 — `layers-tenancy` — and source-list growth on existing contracts; no new contracts in S10-S12 because the SQLAlchemy-in-tenancy.application question resolved against a new contract per D36 reasoning).
- **AST/parser enforcement layer growth:** S10 added `tests/_enforcement/test_no_plaintext_in_state.py`, S11 added `tests/_enforcement/test_host_port_bindings.py`, S12 added `tests/_enforcement/test_dockerfile_workspace_coverage.py`. Three new structural enforcement tests across the package — promotion candidates from comment-level rules to parser-level invariants.
- **Decision velocity:** 6 D-entries across 4 sessions. Lower density than P2's 16, reflecting that P3 implements architecture P2 laid down rather than introducing new commitments.
- **Migration runner failure-mode reproduced cleanly:** stop tenant-b → fail at tenant-b → restart tenant-b → resume from tenant-b. Per-tenant transactional with retry behaves as D36 frames it.

- **D13 breach acknowledged at P3 open and recovered to a deferred-decisions item.** Adding a tenant beyond the test set in P3 requires editing Compose (Option A topology). Recovery path: instance creation as configuration lands when production deployment context arrives and infrastructure-as-code is real. The breach is named in the P3 current-package block and inherited by P4+.
- **The env-passthrough pattern bit three times in P3.** S9 `POSTGRES_CONTROL_PLANE_*`, S10 `contexts/tenancy/pyproject.toml` COPY, S11 `POSTGRES_TENANT_A_*`. Promoted to structural enforcement at S12 via the Dockerfile workspace-coverage parser test (third instance is the promotion threshold per S11 reflection).
- **D34's "eight columns" forecast for the registry table corrected inline to eleven.** The architectural commitment ("persist the wire shape, AAD bound, no plaintext column") was unchanged; the column count followed once the live `EncryptedField` wire shape was reconciled. Worth flagging the lesson that vendor-shape forecasts in D-entries should be reconciled against live module imports during the writing pass, not after.
- **Alembic bracket form `[NAME]` vs `[alembic:NAME]`.** S10 prompt forecast `[alembic:control_plane]`; the actual cookbook pattern is the bare `[NAME]`. Corrected in `current-package.md` body; D34 retains its forecast wording with the correction noted to preserve the append-only spirit.
- **PyPI SSL cert failure on macOS Python 3.14.** Surfaced at S10's first scheduled-check run; persisted through S11/S12. Operator housekeeping (one-time `Install Certificates.command` or vendored `certifi`); not a runner-design gap.

Recovery: three `docs(p2/retrospective): ...` commits ahead of S9 proper. Lesson promoted: future briefs grow a "state at session open" verification step, and framing conversations produce a "Pre-S<N> housekeeping" section listing carryover commits.

## Durable lessons

- **The structural-promotion threshold is the third instance.** Once a comment-level rule has bitten three times across the package (env-passthrough at S9/S10/S11, host-port-binding at S5/S10/S11), the cost of a parser-level structural test is justified by the recurrence. Earlier promotion is premature; later promotion is recurring drift.
- **Verification-list growth is a per-session discipline, not a one-time charter writeup.** S9's verification list named git-clean + compose project + `make up` + pytest + lint. S10 grew it with image-rebuild verification; S11 grew it with image-rebuild-plus-recreate; S12 grew it with `make seed-tenants && make migrate` against a clean stack. Each growth came from a session-open failure mode the prior list did not catch.
- **Pre-write reconciliation is non-negotiable for sessions touching specific vendor APIs.** S9 pgvector digest, S10 SQLAlchemy 2.0 dialects + Alembic bracket form + cryptography InvalidTag, S11 SQLAlchemy 2.0 async pool sizing + Alembic programmatic invocation, S12 OTel multi-tenant attribute conventions + SQLAlchemy 2.0 `with_for_update`. Each reconciliation pass produced real corrections; none were memory-only.
- **Housekeeping items in the carryover list of `current-package.md` are themselves a signal that something is owed across the package boundary.** A session that opens with carryover items should land them in commit 1 of the new package rather than allow the carryover bullet to roll forward. Asymmetric: runtime-substrate housekeeping (directory names, port bindings, image digests) lands as a separate small commit before substantive work; documentation-shaped housekeeping folds into the next substantive commit.
- **Defense-in-depth around plaintext credential leakage is structural, not advisory.** D34's three-control structure (logging filter + AST test + tenant-isolation tests) lands in the same commits as the integration that first writes plaintext. The trigger is structural and detectable: any `AnnAssign` in an adapter `__init__` whose annotation references a domain value object whose docstring contains "MUST NOT PERSIST" is a candidate for the AST forbidden-types set.
- **Per-destination hash chains are forced by D32's instance independence.** A single global chain across destinations would require cross-database hash coordination that contradicts the per-tenant Postgres topology. The per-destination commitment in D35 follows from D32, not from a fresh design choice.

## Convention changes

- **Pre-write vendor-doc reconciliation as a named section at session open** for any session touching specific vendor APIs. Pattern from S6/S7 promoted to a per-session-prompt section in P3.
- **State-at-open verification as a named section at session open** with explicit checks (git status, compose project, `make up`, pytest, lint, scan, sbom, image-rebuild, `make migrate`). The list grows as session-open failure modes surface.
- **Three-instance promotion threshold for comment→AST/parser-level structural tests.** Promoted from S11 reflection to standing convention via S12 commit 7.
- **Carryover items in `current-package.md` are signals to land in commit 1 of the inheriting session,** asymmetric by housekeeping shape (runtime-substrate vs documentation).
- **Browser interactive verification mid-session** rather than at session close for any UI-bearing acceptance criterion. P2's convention from S4 reinforced through P3.
