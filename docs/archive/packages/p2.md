## Architectural commitments shipped

Sixteen D-entries (D15-D30). The structural ones:

- **D16, D17.** Bounded contexts at top level, hexagonal within, cross-context communication via published query APIs and domain event bus, enforced by import-linter.
- **D19.** Configuration through `vadakkan/config/` exclusively, AST test enforces no `os.getenv` outside.
- **D20-D22.** TLS through config layer, envelope encryption via `vadakkan/security/crypto.py`, audit as a bounded context with hash-chained append-only storage.
- **D23, D24.** Auth middleware on every endpoint, tenant-isolation tests mandatory for tenant-scoped adapters.
- **D25, D26.** Supply chain hardening (digest pins, SBOM, scheduled monitoring), security events log separately.
- **D27.** OTel as the observability portability boundary, vendor-specific code confined to adapters.
- **D28, D29.** Package named `vadakkan` (decoupled from product name), product renamed Meridian → Vadakkan.
- **D30.** uv workspace topology: root pyproject is the cross-cutting platform package; bounded contexts and apps are workspace members.

## Data the package surfaced

- **Inference throughput:** 19 tokens/sec on local CPU with Qwen 2.5 7B. Eval cycles at P5 will be measured in tens of minutes. Phase 1 close audit input on whether GPU acceleration or hosted inference is required.
- **Test density at S7:** 27 new tests against 6 substantive commits. Dense, correct for structural session. Watchpoint: incremental feature work in P3+ needs its own calibration.
- **Import-linter contracts:** 3 → 14 across P2. Each contract is a future review comment that does not need to be made. False-positive rate (legitimate refactors blocked vs real drift caught) is data P3 will produce.
- **Service count:** 3 → 11 via incremental scaffold (D11). Memory footprint and startup cost compound per service. P3 adds two more Postgres instances.
- **Decision velocity:** 16 D-entries across 4 sessions. High density, correct for a structural package. P3 should land fewer decisions because the architectural commitments are now laid down.
- **Session count drift:** Planned 3, shipped 4. S8 was inserted post-open for the rebrand. Future package framing names candidate insertions at package open.

## Drift surfaced

- **S5 acceptance grep miscalibrated against append-only history.** Caught at S8. Process drift, not architectural drift. Lesson: when append-only restoration is in scope, criteria name expected residue by location and decision number, not by count.
- **S6 LiteLLM-OTel friction.** Caught by pre-write vendor-doc reconciliation. Architecture absorbed cleanly via D27. The discipline of pre-write reconciliation worked.
- **S7 browser verification operator-action-remaining at session close.** Standing rule from S4 lands as routine practice in P3+: UI-bearing acceptance criteria fold into the body of the session, not the close.
- **S8 directory typo (`Vadkkan` vs `Vadakkan`)** marked operator housekeeping, embedded in compose project naming. Folded into first P3 session for correction.

## Durable lessons

- Vendor-doc reconciliation as a pre-write step in any session touching specific vendor APIs, not memory-based prompt drafting. Standing practice.
- Browser verification as the acceptance gate for any UI-bearing surface. Standing rule, now folded mid-session rather than session-close.
- Append-only discipline. The decisions log and session log are appended, never edited. The rebrand session was the precedent.

## What this package leaves to P3

Tenancy primitives: tenant registry, per-tenant database connections, migration runner, audit log table real adapter. The architectural commitments load-bearing for P3:

- Database-per-tenant (D1) goes from architectural commitment to running implementation. Topology choice: separate Postgres instances per tenant, with the connection-resolution layer in `vadakkan/config/` reading the registry, decrypting credentials via `vadakkan/security/crypto.py` (D21), and constructing per-tenant connections.
- Jurisdiction (D12) as a column on every per-tenant table from inception.
- Tenant onboarding as configuration (D13) via an idempotent provisioning workflow.
- Audit context (D22) real adapter writing to tenant-scoped append-only Postgres with hash chaining.
- Tenant-isolation tests (D24) populated heavily because P3 introduces tenant-scoped data.
- Configuration scope follows tenant agency. Per-tenant configuration covers what the tenant has agency over (jurisdiction, identity federation, classification policy, model endpoints, retention). Platform configuration covers the rest.

## Convention changes

- **Package-level archive at every package close from P2 forward.** Cost calibration revisited at Phase 1 close. P1 retains no archive (audit-time-only convention applied at P1 close); P2 forward writes to `docs/archive/packages/pN.md`.
- **Scheduled supply-chain checks beyond the Langfuse precedent.** Cadence buckets: monthly batch (Langfuse, OTel-instrumentation family, LiteLLM, FastAPI/Uvicorn), quarterly (Pydantic chain), annual (import-linter). Implementation in `ops/scheduled_checks.yaml`.
