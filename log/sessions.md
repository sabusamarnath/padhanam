## Session 4 (Package P2)
- Produced: Charter — D15 (Qwen 2.5 7B default dev model), D16 (bounded contexts on hexagonal with import-linter), D17 (query APIs for reads, event bus for state changes), D18 (shared Redis dev / config-swap path for prod), bounded-contexts principle under Architectural, P2 current-package block. Compose — five new services pinned to SHA256 digests (langfuse-db, clickhouse 24.12, minio chainguard, langfuse-web 3.172.0, langfuse-worker 3.172.0); existing redis shared with Langfuse via DB 1. Caddyfile — `/langfuse/*` handle_path ahead of catch-all with explicit ordering comment. .env.example — Langfuse-related placeholders; .env (gitignored) — generated secrets, ENCRYPTION_KEY in hex, INIT keys in `pk-lf-`/`sk-lf-` shape. schema.md untouched (Langfuse is an isolated subsystem with its own Postgres + ClickHouse + MinIO).
- Decisions: D15, D16, D17, D18.
- Tests: `make up` brings all 8 containers healthy in <30s on the warm path (langfuse-web flips healthy ~15s after start once Next.js Ready). `curl https://localhost/health` → `200 ok`; `curl https://localhost/` → `503`; `curl https://localhost/langfuse/api/public/health` → `{"status":"OK","version":"3.172.0"}`; `curl -I https://localhost/langfuse/` → `HTTP/2 200`. INIT bootstrap verified: `GET /api/public/projects` with the generated keys returns the `meridian` org and `meridian-dev` project. End-to-end ingestion: POST a `trace-create` to `/langfuse/api/public/ingestion` returned 201; after ~8s the trace was retrievable via `GET /api/public/traces/<id>` (POST → web → Redis DB 1 → worker → ClickHouse → queryable). `make down` removes all containers and the project network. Browser-driven login at `https://localhost/langfuse/` not run from CLI — left for operator confirmation.
- Reflection: Langfuse 3's realised deployment footprint is six services: langfuse-web, langfuse-worker, langfuse-db (Postgres for metadata), ClickHouse (trace data plane), Redis (shared with the existing Meridian instance via DB number per D18), and MinIO (S3-compatible event and media blob storage). The trace store moved off Postgres into ClickHouse in v3. This is consequent to D7 (self-hosted Langfuse 3); the alternative paths (Langfuse Cloud, different trace store) contradict either local-first or D7 itself. Footprint accepted. Watchpoint for Phase 1 close audit: if self-hosted observability tooling becomes a meaningful drag on operator time during Phase 1, that is data feeding the Phase 2 pivot decision. The original S4 prompt drafted by Claude.ai assumed a two-service Langfuse deployment, which Claude Code caught at the pre-write topology check; logged as a prompt-quality finding for future vendor-deployment prompts. Path-based routing under `/langfuse/*` worked at the HTML level but failed in the browser: Langfuse is a Next.js app whose `/_next/*` asset URLs only respect a basePath set at build time via `NEXT_PUBLIC_BASE_PATH`, which would require building the image from source and would defeat the upstream-digest pinning landed this session. Switched to subdomain serving on `langfuse.localhost` instead; mkcert reissued with both SANs, Caddyfile gained a second site block, `*.localhost` resolves to 127.0.0.1 on macOS natively. This sets the pattern for future UI services (Keycloak as `auth.localhost`, Meridian app UI as `app.localhost`); UI services on subdomains, APIs on `localhost` paths. Headless `LANGFUSE_INIT_*` worked as documented — org/project/user/keys created on first boot with the .env-supplied values. Two other non-documented issues: Next.js binds to the container hostname interface by default rather than `0.0.0.0`, so `HOSTNAME=0.0.0.0` had to be added to both langfuse services; and busybox wget races IPv6 resolution of `localhost` against an IPv4-only listener, so the healthcheck uses `127.0.0.1` explicitly. Both are noted in the compose commit message. Flat `.env` strain is now visible — the file gained 18 Langfuse-related vars in a single session and is approaching the point where per-service `env_file:` blocks would be more legible than one growing file; the right answer is still to wait for S5's `platform/config/` and Pydantic Settings rather than introduce an interim structure that gets ripped out a session later. Shared Redis behaved cleanly via `REDIS_CONNECTION_STRING=redis://redis:6379/1` (Langfuse 3 supports the connection-string form alongside the discrete HOST/PORT/AUTH triple); no key-namespace collision risk observed since Langfuse uses prefixed keys (`bull:`, `langfuse:`). Charter writeup of D16/D17/D18 surfaced no need for adjustment before S5; the decisions feel proportionate to where the codebase actually is. schema.md correctly left untouched — Langfuse owns its own data plane.
